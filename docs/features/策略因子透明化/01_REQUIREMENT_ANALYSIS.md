# 策略因子权重透明化与可解读体验

> 状态：草案 → 已增强
> 最后更新：2026-04-28
> 关联代码：`src/core/screener_engine.py`, `src/core/strategy_signal_extractor.py`, `src/core/strategy_optimizer.py`, `apps/dsa-web/src/pages/ScreenerPage.tsx`

## 背景

当前系统存在以下问题：

1. **黑盒打分**：用户在选股结果中看到一个 "综合评分 85.3" 但完全不知道这个分数是怎么来的——是换手率高导致的？还是技术形态好？还是 PE 低？
2. **权重不可见**：26 个策略各有权重，但这些权重如何影响选股结果，用户完全看不到
3. **回测反馈不可解释**：回测后权重调整了哪些策略？为什么调？"来源"栏写一个"回测"或"IC"仍然是黑盒——用户想知道的是**为什么回测结果导致这个策略权重被上调 30%**。
4. **阿尔法系统更黑**：参数化因子（如 `decline_threshold`, `trend_score`）完全不可见
5. **"来源"标签本身就是黑盒**：`来源=回测` 没有告诉用户"在最近 50 次验证中，底部放量策略触发的 15 次信号里 10 次盈利，平均收益 +4.2%，因此权重从 0.60 提升到 0.72"

## 目标

- **目标 1**：选股结果中每只股票展示"得分拆解"图表，让用户看到每个策略贡献了多少分
- **目标 2**：提供策略权重一览表，展示当前权重、基础权重、最近调整变化
- **目标 3**：回测反馈后展示权重调整的 before/after 对比及原因
- **目标 4**：阿尔法系统中展示因子参数的值和调整范围
- **目标 5**：提供"问股式"自然语言解释，把技术指标翻译成普通人能理解的语言
- **目标 6（新增）**：LLM 驱动的权重变化解读——不只是"来源=回测"，而是解释**回测里发生了什么导致这个策略被加分/降权**

## 方案设计

### 核心架构

```
选股完成
  │
  ├──→ 结果列表：每只股票展示「得分拆解」按钮
  │       └──→ 展开后显示：
  │             ├─ 基础评分 (换手/动量/估值/市值各几项)
  │             ├─ 策略分组得分 (趋势/反转/形态度量等大类)
  │             └─ 顶部触发的具体策略信号
  │
  ├──→ 权重面板：侧栏或弹窗
  │       ├─ 当前生效权重 vs 基础权重
  │       ├─ 最近优化记录 (日期/原因/变化)
  │       ├─ [NEW] LLM 解读卡片：每条权重变化的自然语言解释
  │       └─ 当前市场状态 (上涨/下跌/震荡) 及对权重的影响
  │
  └──→ 用户操作：点某只股票 → AI 问股式解读
          └─ "这只股票因为换手率适中(3.2%)、PE合理(25倍)获得基础分70分
             同时触发了多头趋势信号(12分权重)和均线金叉(8分权重)
             综合评分85.3，在当前250只候选股中排名第1"
```

### 数据流

```
GET /api/v1/screener/weights/interpretation
  → WeightInterpretationService.generate(changes, backtest_data)
    → 查询每个策略的回测表现 (胜率/平均收益/样本量)
    → 组装 LLM prompt：策略名 + 权重变化 + 回测数据
    → LLM 生成自然语言解读
  → 返回结构化解读列表

LLM Prompt 设计：
"""
你是一个量化策略分析师。请根据以下数据，用通俗易懂的中文解释
为什么某个策略的权重被调整了。

策略: 底部放量 (bottom_volume)
权重变化: 0.60 → 0.72 (+20%)
调整来源: 回测反馈
回测数据:
  - 最近30次分析中触发该信号15次
  - 其中10次盈利 (胜率66.7%)
  - 平均持仓收益 +4.2%
  - 最大单次收益 +12.5%，最大亏损 -3.1%
  - 策略胜率排名: 3/26

请用2-3句话解释：
1. 为什么这个策略被加分
2. 这个调整幅度是否合理
3. 有什么风险需要注意
"""
```

### LLM 权重解读服务设计

```
┌─ WeightInterpretationService ────────────────────┐
│                                                    │
│  generate(                                         │
│    changes: List[WeightChange],                    │
│    backtest_context: Dict[str, StrategyPerf]       │
│  ) → List[WeightInterpretation]                    │
│                                                    │
│  每一条 WeightChange 的解读流程:                     │
│    1. 从 BacktestService 查询该策略的:              │
│       - 总信号数 / 胜率 / 平均收益                  │
│       - 在全部策略中的排名                          │
│       - 最近表现趋势 (上升/下降)                    │
│                                                    │
│    2. 从 StrategyOptimizer 查询:                    │
│       - adjustment_reason (已有)                    │
│       - train/val win_rate (已有)                   │
│       - degradation_ratio (已有)                    │
│                                                    │
│    3. 组装 LLM prompt → 生成解读                    │
│       ├─ 高胜率策略被加分:                           │
│       │  "底部放量策略在最近30次回测中胜率达67%，     │
│       │   平均盈利+4.2%，表现位列26个策略中第3名。    │
│       │   因此权重从0.60上调至0.72(+20%)。           │
│       │   但需注意该策略在震荡市中表现不稳定，         │
│       │   建议市场转弱时手动降权。"                   │
│       │                                            │
│       ├─ 低胜率策略被降权:                           │
│       │  "一阳三阴策略最近30次回测胜率仅33%，         │
│       │   平均亏损-2.1%，且训练集与验证集的          │
│       │   表现比值0.72 < 0.8阈值，存在过拟合风险。    │
│       │   因此权重从0.50下调至0.35(-30%)。           │
│       │   建议暂时降低该策略在选股中的影响力。"       │
│       │                                            │
│       └─ 无变化策略:                                │
│          "多头趋势策略样本不足（仅8次回测），         │
│           暂维持基础权重1.00不变。                   │
│           建议积累更多数据后再评估调整。"             │
│                                                    │
│    4. 缓存结果，1小时内不重复请求LLM                 │
│                                                    │
└────────────────────────────────────────────────────┘
```

### LLM 解读数据来源

| 解读要素 | 数据来源 | 说明 |
|---------|---------|------|
| 策略胜率 | `backtest_repo.get_summary(scope="strategy", code=name)` | 策略级回测汇总 |
| 平均收益 | 同上 `avg_simulated_return_pct` | 触发信号后的平均N日收益 |
| 策略排名 | 26个策略按 `win_rate_pct` 排序 | 相对表现 |
| 调整幅度 | `StrategyOptimizer.get_optimization_details()` | before/after 对比 |
| 过拟合风险 | `OptimizationResult.degradation_ratio` | 验证集/训练集胜率比 |
| IC 有效性 | `AlphaEvaluator.compute_factor_ic()` | 因子信息系数 |
| 市场环境 | `MarketRegimeDetector.detect()` | 当前市场状态 |
| 样本量 | `StrategyPerformance.sample_count` | 触发次数 |

### LLM 解读的触发时机

| 触发场景 | 行为 |
|---------|------|
| 用户打开权重面板 | 如果解读缓存未过期，展示已有解读；否则生成新解读 |
| 回测反馈完成 | 自动生成所有变化策略的解读 |
| 用户点击"刷新解读" | 强制重新生成 |
| Alpha 优化完成 | 同上 |

### 解读内容缓存策略

- 解读内容在权重未变化时缓存 24 小时
- 权重变化后自动失效
- 缓存键：`{strategy_name}:{optimized_weight}:{optimization_date}`

## 风险与约束

- **风险1**：分数拆解API需要单只股票查询，10只股票会产生10次请求 → 用批量API (一次请求返回所有股票拆解)
- **约束1**：前端JSON体积 — 500只候选股的完整拆解不可行 → 仅Top-N拆解详细，其余摘要
- **风险2（新增）**：LLM解读可能不准确或遗漏关键信息 → LLM输出需要标明"由AI生成，仅供参考"，且结构化数据（胜率、收益）直接显示数字，LLM仅做解读
- **约束2（新增）**：LLM调用成本 — 26个策略全量解读约需1-2次LLM调用 → 仅对**有权重变化的策略**生成解读，无变化策略只标注原因
