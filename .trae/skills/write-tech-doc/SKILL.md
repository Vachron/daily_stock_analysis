# Write Tech Doc

为项目编写技术文档，确保新 agent 能通过 `KNOWLEDGE_INDEX.md` 快速定位核心知识与代码。

## 核心原则

1. **先读后写**：编写任何文档前，必须先阅读 `KNOWLEDGE_INDEX.md` 和相关已有文档，避免重复或矛盾。
2. **位置规范**：技术文档统一放在 `docs/` 目录下，文件名使用小写+连字符，如 `docs/factor-optimization.md`。
3. **索引联动**：每新增一篇 `docs/` 下的文档，必须在 `KNOWLEDGE_INDEX.md` 对应分类中添加索引条目。
4. **语言一致**：文档默认使用中文，与项目现有文档风格保持一致。代码片段中的注释与文件语境保持一致。
5. **AGENTS.md 优先**：文档内容不得与 `AGENTS.md` 硬规则冲突；若发现冲突，以 `AGENTS.md` 为准并标注差异。

## 文档模板

### 设计方案类文档

```markdown
# [方案名称]

> 状态：草案 / 评审中 / 已批准 / 已实施
> 最后更新：YYYY-MM-DD
> 关联代码：`src/core/xxx.py`

## 背景

为什么要做这件事，当前痛点是什么。

## 目标

- 目标1
- 目标2

## 方案设计

### 核心架构

[架构图或流程图]

### 关键模块

| 模块 | 文件 | 职责 |
|------|------|------|
| ... | ... | ... |

### 数据流

[数据流转路径]

## 风险与约束

- 风险1：描述 + 缓解措施
- 约束1：描述

## 验证方式

- [ ] 验证项1
- [ ] 验证项2

## 实施计划

| 阶段 | 内容 | 依赖 |
|------|------|------|
| 1 | ... | ... |

## 参考链接

- [相关资料]
```

### 防范规范类文档

```markdown
# [规范名称]

> 最后更新：YYYY-MM-DD
> 适用范围：[涉及的代码模块]

## 问题描述

[要防范的问题是什么，为什么重要]

## 风险清单

| 风险项 | 严重度 | 触发条件 | 现有代码位置 |
|--------|--------|----------|-------------|
| ... | 🔴/🟡/🟢 | ... | `file.py:L123` |

## 防范规则

### 规则1：[名称]

- **要求**：[必须做什么]
- **检测方法**：[如何自动检测违规]
- **例外**：[什么情况下可以豁免]

## 检查清单

- [ ] 检查项1
- [ ] 检查项2
```

## 操作流程

1. 阅读 `KNOWLEDGE_INDEX.md`，确认文档是否已存在或需要新增。
2. 阅读相关源代码，确保文档内容与实际实现一致。
3. 按模板编写文档，保存到 `docs/` 目录。
4. 更新 `KNOWLEDGE_INDEX.md` 索引。
5. 如果文档涉及用户可见能力或 API 行为变化，同步更新 `docs/CHANGELOG.md` 的 `[Unreleased]` 段。
6. 交付说明中列出：新增了哪些文档、更新了哪些索引、是否有未同步的英文文档。
